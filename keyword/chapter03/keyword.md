# SOLID원칙이란?

SOLID는 객체지향 설계를 더 유연하고 유지보수 가능하게 만들기 위한 5가지 설계 원칙이다.

Spring을 공부할 때 SOLID가 중요한 이유는, Spring의 DI, IoC, 인터페이스 기반 설계, 느슨한 결합 같은 철학이 이 원칙들과 자연스럽게 연결되기 때문이다.

## 1. SRP - 단일 책임 원칙

하나의 클래스는 하나의 책임만 가져야 한다는 원칙이다.

즉, 클래스가 변경되어야 하는 이유는 하나여야 한다.

예를 들어 회원가입 서비스가 회원 검증, 이메일 발송, DB 저장, 로그 기록까지 전부 담당하면 책임이 너무 많아진다.

이 경우 역할을 나누면 수정 범위가 줄고 유지보수가 쉬워진다.

## 2. OCP - 개방 폐쇄 원칙

확장에는 열려 있고, 수정에는 닫혀 있어야 한다는 원칙이다.

기존 코드를 많이 바꾸지 않고도 기능을 추가할 수 있어야 한다.

Spring에서 인터페이스와 구현체를 분리하고, DI로 구현체를 갈아끼우는 구조가 대표적인 예시다.

## 3. LSP - 리스코프 치환 원칙

자식 클래스는 부모 클래스나 인터페이스를 대체할 수 있어야 한다는 원칙이다.

즉, 구현체를 바꿔도 사용하는 쪽의 기대가 깨지면 안 된다.

## 4. ISP - 인터페이스 분리 원칙

하나의 큰 인터페이스보다, 목적에 맞는 작은 인터페이스 여러 개로 나누는 것이 좋다는 원칙이다.

클라이언트가 자신이 사용하지 않는 메서드에 의존하지 않도록 하기 위함이다.

## 5. DIP - 의존 역전 원칙

구체적인 구현 클래스가 아니라 추상화(인터페이스)에 의존해야 한다는 원칙이다.

Spring의 DI와 가장 직접적으로 연결되는 원칙이다.

예를 들어 OrderService가 KakaoPayService에 직접 의존하는 것이 아니라 PayService 인터페이스에 의존하면, 구현체를 더 쉽게 교체할 수 있다.

## 정리

SOLID는 단순 암기용 규칙이 아니라,

**변경에 강하고 테스트하기 쉬운 구조를 만들기 위한 설계 기준**이다.

특히 Spring에서는 DIP, SRP, OCP를 많이 체감하게 된다.

# DI란?

## 1. What it is

DI는 Dependency Injection, 한국어로는 의존성 주입이다. 의존성 주입..? 잘 와닿지 않지만 객체에 빗대어 표현하면 예를들어 OrderService가 결제를 하려면 PaymentService가 필요할 수 있고, 회원 조회를 하려면 MemberRepository가 필요할 수 있다.

즉, 객체가 혼자 일하려면 보통 다른 객체가 필요로하고 종종 다른 객체에 의존한다.

DI는 이때 객체가 필요한 의존성을 직접 만들지 않고, 외부에서 전달하는 방식을 말한다.

정리하면

- 객체가 자기 의존성을 직접 생성하지 않는다.

## 외부가 적절한 의존성을 연결해준다.

## 2. Why it exists

DI가 필요한 이유는, 객체가 필요한 걸 직접 만들기 시작하면 **결합도가 높아지기 때문**이다.

예를 들어 이런 구조를 보자.

```java
public class OrderService {
    private final KakaoPaymentService paymentService = new KakaoPaymentService();
}
```

이 경우 `OrderService`는 단순히 “결제 기능”에 의존하는 게 아니라,

아주 구체적으로 `KakaoPaymentService`라는 구현에 묶인다.

이렇게 되면 문제가 생긴다.

### (1) 구현체 교체가 어렵다

나중에 `TossPaymentService`로 바꾸고 싶으면

`OrderService` 코드를 수정해야 한다.

### (2) 테스트가 어렵다

테스트용 가짜 구현(mock, stub)을 넣고 싶어도

내부에서 이미 구현체를 직접 만들고 있어서 바꾸기 어렵다.

### (3) 책임이 섞인다

`OrderService`의 본래 책임은 주문/결제 흐름을 조율하는 것인데,

구현체 선택과 생성까지 책임지게 된다.

즉 DI는 이런 문제를 줄이기 위해 존재한다.

핵심 목적은:

- 결합도 낮추기
- 교체 가능성 높이기
- 테스트 쉽게 만들기
- 책임 분리 명확히 하기

---

## 3. Where it fits

DI는 Spring에서 **IoC를 실현하는 대표 메커니즘**이다.

관계를 다시 보면:

- **IoC** = 제어권을 외부로 넘긴다
- **DI** = 그 제어권 이동을 객체 관계 수준에서 구현한다

즉 IoC가 상위 원리라면,

DI는 “그러면 객체를 실제로 어떻게 연결할 건데?”에 대한 대답이다.

Spring에서는 보통:

- Bean을 생성하고
- 필요한 의존성을 찾아서
- 생성자나 세터 등을 통해 넣어준다

이 과정이 일어나고,

이게 바로 DI가 실제로 동작하는 장면이다.

## 5. 컴파일 시점 / 런타임 시점

### 컴파일 시점

컴파일 시점에는:

- 클래스 정의가 컴파일되고
- 생성자, 필드, 어노테이션 정보가 남는다

예를 들어 생성자 주입 구조를 작성해두면

“이 클래스는 이 의존성이 필요하다”는 정보가 코드 형태로 표현된다.

하지만 실제 주입은 아직 일어나지 않는다.

---

### 런타임 시점

런타임에 스프링 컨테이너가:

1. Bean 후보를 찾고
2. 객체를 생성하고
3. 어떤 의존성이 필요한지 확인하고
4. 적절한 Bean을 찾아 주입한다

즉 DI의 실제 실행은 주로 **런타임 시점**에 일어난다.

이건 IoC와 마찬가지로 매우 중요해.

DI는 소스코드 문법이 아니라, **런타임 구조**로 이해해야 한다.

---

## 6. 예시로 보면

### DI가 없는 구조

```java
public class CoffeeMachine {
    private final RegularCoffeeBean coffeeBean = new RegularCoffeeBean();
}
```

여기서는 CoffeeMachine이 어떤 원두를 쓸지 직접 결정한다.

즉 구현체 선택과 생성 책임까지 자기 안에 갖고 있다.

---

### DI가 있는 구조

```java
public class CoffeeMachine {
    private final CoffeeBean coffeeBean;

    public CoffeeMachine(CoffeeBean coffeeBean) {
        this.coffeeBean = coffeeBean;
    }
}
```

이 구조에서는 CoffeeMachine은 그냥

“나는 CoffeeBean이 필요해”라고만 말한다.

어떤 구현을 줄지, 언제 만들지, 어떤 인스턴스를 연결할지는 외부가 결정한다.

이게 DI의 핵심이다!!

---

## 7. Spring에서 DI를 적용하는 방식

워크북도 정리하듯, 스프링에서 자주 나오는 주입 방식은 세 가지다.

### 1) 생성자 주입

가장 권장되는 방식이다.

이유:

- 필수 의존성을 강제할 수 있다
- `final` 사용 가능
- 의존성이 코드에 명확하게 드러난다
- 테스트하기 쉽다

### 2) 수정자(Setter) 주입

선택 의존성에 적합하다.

하지만 객체가 불완전한 상태로 생성될 수 있다는 문제가 있다.

### 3) 필드 주입

코드는 짧지만, 의존성이 숨겨지고 테스트가 어려워서 실무에서는 덜 권장된다.

즉 Spring에서 DI를 배울 때 실제로 가장 중요한 것은

**생성자 주입이 왜 표준처럼 권장되는가**를 이해하는 것이다.

---

## 8. 실무적 관점

실무에서 DI를 이해한다는 것은 단순히 `@Autowired` 문법을 아는 게 아니다.

진짜 중요한 건:

- 이 클래스가 어떤 의존성을 가지는지 명확한가?
- 구현체에 직접 묶이지 않았는가?
- 테스트용 대체 구현을 넣기 쉬운가?
- 객체 생성 책임이 외부화되어 있는가?

좋은 DI 구조는 결국

**변경 가능성, 테스트성, 유지보수성**으로 이어진다.

---

사람이 검수할 때는 이런 질문을 해야 한다.

# IoC란?
## 1. What it is

IoC는 inversion of Control, 번역하면 제어의 역전이다.

말이 조금 추상적으로 들리는데, 핵심은 간단하다.

원래 프로그램을 만들 때는 보통 개발자가 직접:

- 객체를 만들고
- 객체끼리 연결하고
- 어떤 순서로 실행할지 정하고
- 필요한 시점에 메서드를 호출한다!.!

즉 제어권이라는 것을 생각하지도 않게, 당연히 개발자 코드에 있다.

그런데 IoC에서는 이 제어권의 일부가 외부 컨테이너나 프레임워크로 넘어간다.

즉 개발자는 **무엇을 할지**를 정의하고, **언제 만들고 어떻게 연결하고 어떤 순서로 실행할지**의 상단 부분은 프레임워크가 담당한다.

즉, **IoC**는 객체 생성, 생명주기, 실행 흐름의 제어권을 개발자가 아닌 외부 시스템이 가지는 구조이다.

> 객체 생성을 가진다..? 생성되는 타이밍을 말하는 것인가? 근데 결국에 개발자가 선언해야 생성되는거 아닌가? 생명주기에 저런 생성은 포함안되나?
>

## 2. Why it exists

IoC가 필요한 이유는 프로그램 규모가 커질수록 개발자가 모든 걸 직접 제어하는 방식이 금방 복잡해지기 때문이다.

### (1) 객체 관계가 복잡해진다

작은 프로그램에서는 객체를 직접 `new`로 만들고 연결해도 큰 문제가 없다.

하지만 애플리케이션이 커지면:

- 서비스가 레포지토리를 쓰고
- 레포지토리가 다른 컴포넌트를 쓰고
- 인증, 로깅, 설정 객체까지 섞이고
- 객체 수가 급격히 늘어난다

이때 개발자가 모든 객체 생성과 연결을 수동으로 관리하면,

관계가 꼬이고 변경이 어려워진다.

즉 IoC는 **객체 그래프 관리의 복잡성을 줄이기 위해** 필요하다.

---

### (2) 구조를 일관되게 유지하기 위해

프레임워크는 보통 “이 계층은 이런 역할을 하고, 저 계층은 저런 역할을 한다”는 구조를 제공한다.

그런데 개발자가 객체 생성과 흐름 제어를 다 제멋대로 하면, 프레임워크가 제공하는 구조적 장점이 사라진다.

IoC는 제어권을 프레임워크 쪽으로 가져가면서

애플리케이션이 **일정한 구조와 규칙 안에서 움직이도록** 만든다.

즉 IoC는 단순한 편의 기능이 아니라 **구조 강제 장치**이기도 하다.

---

### (3) 변경 용이성과 테스트성을 높이기 위해

개발자가 직접 객체를 만들고 흐름을 엮으면,

구현체를 바꾸거나 테스트 대역(mock, stub)을 넣기 어려워진다.

반대로 IoC 구조에서는:

- 객체 생성이 한곳에 모이고
- 의존성 연결이 외부화되고
- 필요한 구현을 바꿔 끼우기 쉬워진다

즉 IoC는 **결합도를 낮추고 테스트하기 쉬운 구조**를 만드는 데 기여한다.

---

### (4) 프레임워크가 구조를 주도하기 위해

프레임워크의 특징은 **프레임워크는 실행 구조를 주도한다**는 것이다.

IoC는 바로 그 프레임워크적 성격이 실제 코드 구조에 반영된 형태다.

- 프레임워크는 구조를 제공한다
- 구조를 제공하려면 흐름과 객체 생명주기를 어느 정도 통제해야 한다
- 그래서 IoC가 필요하다

즉 IoC는 프레임워크가 “진짜 프레임워크답게” 동작하기 위한 핵심 원리다.

---

## 3. Where it fits

IoC는 Spring에서 **가장 바깥 철학이자 구조 원리, 철학**에 해당한다.

즉 DI, Bean, ApplicationContext 같은 것보다 **더 상위 개념**이다.

관계를 계층적으로 보면 이렇게 이해하면 돼.

### 상위 원리

- **IoC**: 제어권을 외부로 넘긴다

### 그 원리를 구현하는 메커니즘

- **DI**: 의존성을 외부에서 주입한다
- **Bean 관리**: 객체를 컨테이너가 생성/보관/조회한다
- **ApplicationContext**: 그 컨테이너 역할을 수행한다

즉 IoC는 “방법”이라기보다 **설계 원리**에 더 가깝고,

DI와 Bean 컨테이너는 그 원리를 Spring에서 실제로 구현하는 도구다.

---

### 웹 요청 흐름에서도 IoC가 보인다

IoC는 객체 생성 쪽에서만 나타나는 게 아니다.

웹 요청 흐름에서도 드러난다.

예를 들어 Spring MVC에서 개발자는 컨트롤러 메서드를 작성하지만,

그 메서드를 직접 호출하지 않는다.

요청이 들어오면 **DispatcherServlet이 적절한 시점에 그 메서드를 호출**한다. 요청이 DispatcherServlet으로 들어오고, DispatcherServlet이 적절한 Controller를 찾아 실행하는 에를 생각할 수 있다.

즉 여기서도 제어권은 개발자 코드가 아니라 프레임워크에 있다.

그래서 IoC는:

- 객체 관리
- 의존성 연결
- 웹 요청 흐름
- 공통 처리 구조

전반에 걸쳐 나타나는 **프레임워크적 제어 구조**라고 볼 수 있다.

---

## 4. How it differs from nearby concepts

IoC는 추상적이라서 근처 개념과 비교해야 윤곽?이 잡히기에 머리속으로 그림을 그리며 비교해보자~

### IoC vs DI

이 둘은 가장 많이 헷갈린다.

### IoC

- 더 큰 개념
- 제어권이 외부로 넘어간다는 원리
- 객체 생성, 생명주기, 호출 흐름까지 포함하는 상위 아이디어

### DI

- IoC를 구현하는 대표 방식 중 하나
- 객체가 필요한 의존성을 직접 만들지 않고 외부에서 주입받는 방식

즉:

- **IoC는 원리**
- **DI는 구현 방법**

UMC 워크북도 정확히

“IoC가 제어권을 외부로 넘긴다는 원칙이라면, DI는 IoC를 실현하는 방법 중 하나”라고 설명한다.

---

### IoC vs 단순 객체 생성 위임

가끔 “팩토리에서 객체 만들어주면 그게 IoC 아닌가?”라고 생각할 수 있다.

부분적으로는 맞을 수 있지만, IoC는 단순히 생성만 대신해주는 것보다 더 넓다.

IoC에는 보통 이런 것들이 함께 들어간다.

- 객체 생성
- 객체 생명주기 관리
- 의존관계 연결
- 호출 타이밍 제어
- 공통 흐름 제어

즉 IoC는 단순 생성 위임보다 **구조 전체의 제어권 이동**에 가깝다.

---

### IoC vs Framework generality

프레임워크는 구조를 제공하는 뼈대이다. IoC는 그 프레임워크적 성격을 **구현 수준에서 설명하는 개념**이다.

즉:

- “프레임워크는 구조를 주도한다”는 일반론이 있고
- “그 구조 주도가 코드 수준에서 나타난 것”이 IoC다

그러니까 IoC는 프레임워크 일반론의 **구체화된 구조 원리**라고 보면 된다.

---

### IoC vs Service Locator

이건 조금 더 딮한? 비교이다.

Service Locator 패턴은 객체가 필요한 의존성을 **어떤 전역 저장소/레지스트리에서 찾아오는 방식**이다.

겉보기엔 외부에서 관리되는 것 같지만, 실제로는 **객체 자신이 능동적으로 의존성을 찾는다**는 점에서 DI와 다르다.

반면 DI는 객체가 의존성을 직접 찾지 않고 **외부가 넣어준다.**

즉 둘 다 “외부 관리” 느낌이 있지만,

- Service Locator: 객체가 찾는다
- DI: 외부가 넣어준다

그리고 Spring은 주로 DI 쪽이 핵심이다.

---

## 5. 컴파일 시점 / 런타임 시점

### 컴파일 시점

컴파일 시점에는 자바 코드가 `.class`로 변환되고,

어노테이션 정보도 메타데이터로 남는다.

예를 들면:

- `@Component`
- `@Service`
- `@Configuration`
- `@Autowired`

이런 정보가 클래스에 붙어 있는 상태가 된다.

하지만 이 시점에는 아직:

- 빈이 실제로 생성되지도 않았고
- 의존성이 실제로 연결되지도 않았고
- 웹 요청 흐름도 돌지 않는다

즉 **IoC의 핵심 동작은 아직 일어나지 않는다.**

---

### 런타임 시점

애플리케이션이 실제로 실행되면,

Spring이 클래스패스를 스캔하고 어노테이션을 읽어서:

- 빈 후보를 찾고
- 객체를 생성하고
- 의존성을 연결하고
- 웹 애플리케이션이라면 요청 처리 구조까지 구성한다

즉 IoC의 본질적인 동작은 대부분 **런타임 시점**에 일어난다.

좀 더 구체적으로 말하면 런타임에:

1. Spring 컨테이너 시작
2. Bean 정의 수집
3. Bean 생성
4. 의존성 주입
5. 생명주기 관리
6. 웹이면 DispatcherServlet 등 구조 초기화

가 이뤄진다.

즉 IoC는 “컴파일 시 개념”이라기보다

**런타임에서 실현되는 구조 원리**라고 이해하면 좋다.

---

## 6. 예시로 보면

### IoC가 없는 구조

```java

public class OrderService {
    private final PaymentService paymentService = new KakaoPaymentService();
}
```

여기서는 `OrderService`가 직접 `KakaoPaymentService`를 만든다.

즉 제어권이 `OrderService` 안에 있다.

- 어떤 구현체를 쓸지
- 언제 만들지
- 어떻게 연결할지

를 이 클래스가 직접 정한다.

---

### IoC가 있는 구조

```java
@Service
public class OrderService {
    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```

여기서는 `OrderService`가 `PaymentService` 구현체를 직접 만들지 않는다.

어떤 구현을 줄지, 언제 만들지, 어떻게 연결할지는 외부(Spring 컨테이너)가 결정한다.

즉 제어권이 클래스 바깥으로 나갔다.

이게 IoC가 코드 구조에서 보이는 전형적인 예이다.

---

## 7. 웹 요청 흐름에서 IoC를 다시 보면

IoC는 객체 생성에만 있는 게 아니라 아까 말했듯이 웹 요청 처리에서도 나타난다.

예를 들어:

```java
@RestController
public class MyController {
    @GetMapping("/hello")
    public String hello() {
        return "hello";
    }
}
```

개발자는 `hello()` 메서드를 정의만 한다.

그런데 실제로 이 메서드를 “언제, 어떤 요청에서, 어떤 파라미터와 함께” 실행할지는 개발자가 직접 제어하지 않는다.

런타임에 요청이 오면:

- DispatcherServlet이 요청을 받음
- HandlerMapping이 대상 메서드를 찾음
- HandlerAdapter가 실행
- 필요한 파라미터 바인딩
- 그다음 컨트롤러 메서드 호출

이 흐름이 일어난다. 즉 **호출 흐름의 제어권도 프레임워크가 갖는다.**

그래서 IoC는 단순히 “객체 생성만 컨테이너가 대신한다”가 아니라,

**애플리케이션의 구조적 흐름 전반에서 제어권이 외부화된 상태**라고 봐야 한다.

---

## 8. 실무적 관점

실무에서 IoC를 이해해야 하는 이유는,

Spring 코드를 단순 문법 수준이 아니라 **구조 수준**에서 볼 수 있게 해주기 때문이다.

좋은 Spring 코드는 대체로 이런 특징이 있다.

- 객체를 직접 `new`로 만들지 않는다
- 의존성이 코드에 드러난다
- 생명주기 관리가 컨테이너 중심이다
- 흐름 제어를 프레임워크 구조 안에서 한다

반대로 구조가 나쁜 코드는:

- 서비스 안에서 구현체를 직접 생성하고
- 컨트롤러가 모든 걸 직접 호출하고
- 스프링 빈을 우회해서 수동 조립하고
- 프레임워크가 제공하는 흐름을 제멋대로 깨뜨린다

즉 IoC를 이해하면

“이 코드가 정말 Spring 구조를 활용하는지, 아니면 그냥 Spring 문법만 흉내 낸 건지”를 구분할 수 있게 된다.

---

## 9. AI 코드 검수 관점

AI가 Spring 코드를 만들 때 자주 하는 실수는,

겉으로는 Spring을 쓰는 것 같지만 실제로는 IoC 구조를 망가뜨리는 것이다.

예를 들면:

- `@Service` 안에서 구현체 직접 `new`
- 테스트를 위해 쉽게 바꿀 수 없게 강한 결합 생성
- Spring이 관리해야 할 객체를 수동 생성
- Bean 주입 구조보다 편한 즉흥 코드 우선

이런 코드는 돌아갈 수는 있다.

하지만 구조적으로는 Spring의 핵심 장점을 버리는 셈이다.

그래서 사람이 검수할 때는 이런 질문을 해야 한다.

- 이 객체의 생성 책임이 어디에 있지?
- 이 클래스가 구현체를 직접 고정하고 있지 않은가?
- 의존관계가 외부화되어 있는가?
- 프레임워크의 흐름 제어를 존중하고 있는가?

---

## 10. 지금 단계에서 꼭 남기면 좋은 핵심 문장

- IoC는 **제어권을 개발자 코드가 아니라 외부 컨테이너/프레임워크가 가지는 구조**다.
- Spring에서 IoC는 **객체 생성, 의존성 연결, 생명주기 관리, 요청 처리 흐름** 전반에 나타난다.
- DI는 IoC를 실현하는 대표적인 방법이다.
- IoC의 핵심 동작은 주로 **런타임 시점**에 실제로 일어난다.

# 생성자 주입 vs 수정자, 필드 주입 차이는?

Spring에서 의존성을 주입하는 대표적인 방법은 생성자 주입, 수정자(setter) 주입, 필드 주입이 있다.

## 1. 생성자 주입

생성자를 통해 의존성을 주입하는 방식이다.

### 장점

- 객체 생성 시점에 필수 의존성이 반드시 주입된다
- `final` 사용이 가능해 불변성을 보장하기 좋다
- 의존성이 코드에 명확하게 드러난다
- 테스트 작성이 쉽다
- 가장 권장되는 방식이다

### 단점

- 의존성이 너무 많으면 생성자가 길어질 수 있다
  하지만 이 경우는 오히려 클래스 책임이 많다는 신호일 수 있다

---

## 2. 수정자(Setter) 주입

setter 메서드를 통해 의존성을 주입하는 방식이다.

### 장점

- 선택적인 의존성 주입에 사용할 수 있다
- 주입 시점을 비교적 유연하게 조절할 수 있다

### 단점

- 객체가 완전하지 않은 상태로 생성될 수 있다
- 필수 의존성을 강제하기 어렵다
- 잘못하면 NullPointerException 위험이 생긴다

---

## 3. 필드 주입

필드에 바로 `@Autowired`를 붙여 주입하는 방식이다.

### 장점

- 코드가 짧고 간단해 보인다

### 단점

- 의존성이 겉으로 드러나지 않는다
- 테스트가 어렵다
- 객체 설계를 파악하기 어렵다
- 실무에서는 권장되지 않는다

---

## 결론

실무에서는 **생성자 주입이 가장 권장**된다.

이유는 필수 의존성을 강제할 수 있고, 의존성이 명확하게 드러나며, 테스트와 유지보수에 유리하기 때문이다.

## 한 줄 정리

- 생성자 주입: 필수 의존성, 가장 권장
- 수정자 주입: 선택 의존성에 적합
- 필드 주입: 간단하지만 유지보수성과 테스트성이 떨어짐

# AOP란?

AOP는 **Aspect Oriented Programming**, 즉 **관점 지향 프로그래밍**이다.

핵심 비즈니스 로직과 공통 관심사를 분리해서 관리하기 위한 방식이다.

## 왜 필요한가?

여러 클래스에서 반복적으로 들어가는 코드가 있다.

예를 들면:

- 로깅
- 트랜잭션 처리
- 보안 검사
- 예외 처리
- 성능 측정

이런 기능은 핵심 비즈니스 로직은 아니지만 여러 곳에서 반복된다.

이런 코드를 각 클래스에 직접 넣으면 중복이 많아지고 유지보수가 어려워진다.

## AOP의 핵심 아이디어

공통적으로 필요한 기능을 한곳에 모아두고,

필요한 지점에 공통적으로 적용하는 것이다.

즉:

- 핵심 관심사: 회원가입, 주문 처리, 결제
- 공통 관심사: 로깅, 보안, 트랜잭션

이 둘을 분리하는 것이 AOP의 핵심이다.

## Spring에서의 AOP

Spring은 프록시 기반으로 AOP를 적용한다.

개발자는 비즈니스 로직에만 집중하고, 공통 기능은 별도로 관리할 수 있다.

## 장점

- 중복 코드 감소
- 핵심 로직이 더 깔끔해짐
- 공통 기능을 한곳에서 관리 가능
- 유지보수가 쉬워짐

# 서블릿이란?

서블릿(Servlet)은 **자바에서 HTTP 요청을 처리하기 위한 서버 측 프로그램**이다.

웹 브라우저의 요청을 받아서 처리하고, 그 결과를 응답으로 돌려주는 역할을 한다.

## 왜 필요한가?

정적인 HTML만 제공하던 초기 웹과 달리,

사용자 요청에 따라 동적으로 응답을 생성할 필요가 생겼다.

서블릿은 이런 동적 웹 처리를 자바에서 가능하게 해주는 기술이다.

## 기본 역할

- 클라이언트의 HTTP 요청 받기
- 요청에 맞는 로직 실행
- HTTP 응답 생성해서 반환

## 서블릿 컨테이너

서블릿은 혼자 동작하지 않고, **서블릿 컨테이너**가 관리한다.

서블릿 컨테이너는:

- 서블릿 생성
- 초기화
- 실행
- 종료

같은 생명주기를 관리한다.

대표적인 예가 **Tomcat**이다.

## Spring과의 관계

Spring MVC에서는 모든 요청을 직접 개별 서블릿이 처리하는 것이 아니라,

**DispatcherServlet**이라는 프론트 컨트롤러 서블릿이 먼저 요청을 받는다.

그 후 적절한 컨트롤러로 요청을 전달한다.

즉, Spring MVC도 결국 서블릿 기반 위에서 동작한다.
